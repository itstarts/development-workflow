# 工作流交接状态中文化

## 产品问题

开发工作流会在回复末尾展示 requirements 八字段或 requirements、spec、plan 十四字段状态快照。当前字段名和枚举值以英文展示，中文用户需要先理解机器字段及其状态语义，才能判断文档、评审与门禁进度。

这些状态同时承担机器交接职责，不能直接把内部字段和值整体翻译为中文。若用户可见呈现与内部权威状态混为一体，可能破坏既有校验、自动衔接和旧交接输入兼容性。

## 目标与成功衡量

- 用户可见的八字段和十四字段状态快照使用自然、准确的中文字段名与中文状态值。
- 路径、主题、置信度数字等不属于状态枚举的实际内容保持原值，不作无意义翻译。
- 内部权威机器字段、枚举值和门禁判断继续使用既有英文 canonical 契约，不改变其业务语义。
- 同一交接点只展示一个权威的中文状态快照，不同时追加英文状态块。
- 已有英文状态快照继续可作为下游输入，新生成的用户可见快照使用中文。
- 中文映射不能可靠完成时，系统保持原 canonical 快照不变并关闭自动交接，不输出部分中文或回退英文的混合结果。

## 非目标

- 不翻译或重命名内部 YAML 元数据键、脚本参数、JSON 字段、评估字段、校验器字段或其它机器接口。
- 不改变文档批准、独立评审、技术规格门禁、实施门禁或会话路由的既有业务规则。
- 不翻译路径、主题标识符、skill 名称、命令、文件名或代码标识符。
- 不回写或批量改造历史回复、历史评估结果和已有文档中的英文状态快照。
- 不安装到真实 `CODEX_HOME`，不创建用户可见任务，不提交、不发布。
- 不在本 PRD 中确定映射函数、文件组织、渲染架构或具体测试实现。

## 目标用户

- 以中文使用 Codex 开发工作流，需快速判断 PRD、技术规格、实施计划和门禁状态的用户。
- 维护五个工作流 skill 及其交接契约的贡献者。

## 用户场景

### 场景一：查看 PRD 八字段交接

用户完成需求澄清、评审或批准操作后，在回复末尾看到中文八字段快照。用户无需理解 `requirements_user_approval`、`approved` 或 `specification_gate` 等英文术语，即可判断需求文档和技术规格门禁的当前状态。

### 场景二：查看完整十四字段交接

用户完成技术规格或实施计划阶段后，在回复末尾看到中文十四字段快照。需求、技术规格、计划和双门状态使用一致的中文语义，路径和主题仍保持可复制、可执行的原值。

### 场景三：沿用已有英文交接输入

下游工作流收到先前生成的英文八字段或十四字段快照时，仍按既有 canonical 字段和值读取和校验；兼容旧输入不要求用户先手工翻译。

### 场景四：遇到未知但合法的状态

内部 canonical 状态明确为既有的 `unknown` 时，用户可见值显示“未知”。这是正常、可映射的状态，不应被误报为系统故障。

### 场景五：中文映射无法可靠完成

当实现遗漏了新的 canonical 枚举、同一 canonical 状态出现冲突映射，或发生其它确定性映射缺陷时，系统不输出残缺中文快照，也不回退显示英文状态块。系统保留原 canonical 快照和批准、门禁事实，仅暂停依赖该快照的本次自动交接，并报告确定性阻塞；修复后从原 canonical 快照重新生成。

## 范围与产品需求

### 用户可见字段名

八字段和十四字段快照按字段语境使用以下中文名称；同一语义字段在不同阶段保持一致：

| 内部 canonical 字段 | 用户可见字段名 |
| --- | --- |
| `requirements_path` | 需求文档 |
| `requirements_topic` | 需求主题 |
| `requirements_scope` | 需求范围 |
| `understanding_confidence` / `requirements_understanding_confidence` | 需求理解置信度 |
| `understanding_user_confirmation` / `requirements_understanding_confirmation` | 需求理解确认 |
| `requirements_user_approval` | 需求文档用户批准 |
| `requirements_independent_review` | 需求文档独立评审 |
| `specification_gate` | 技术规格门禁 |
| `spec_path` | 技术规格 |
| `spec_user_approval` | 技术规格用户批准 |
| `spec_independent_review` | 技术规格独立评审 |
| `plan_path` | 实施计划 |
| `plan_review_status` | 计划评审状态 |
| `implementation_gate` | 实施门禁 |

### 用户可见枚举值

内部 canonical 值保持英文，用户可见值按字段语境映射：

| 字段语境 | 内部 canonical 值 | 用户可见值 |
| --- | --- | --- |
| 需求范围 | `product` | 产品 |
| 需求范围 | `phase` | 阶段 |
| 需求范围 | `feature` | 功能 |
| 需求理解确认 | `pending` | 待确认 |
| 需求理解确认 | `approved` | 已确认 |
| 需求理解确认 | `unknown` | 未知 |
| 需求文档用户批准 | `pending` | 待批准 |
| 需求文档用户批准 | `approved` | 已批准 |
| 需求文档用户批准 | `unknown` | 未知 |
| 技术规格用户批准 | `pending` | 待批准 |
| 技术规格用户批准 | `approved` | 已批准 |
| 需求文档独立评审 | `pending` | 待评审 |
| 需求文档独立评审 | `approved` | 已通过 |
| 需求文档独立评审 | `unknown` | 未知 |
| 技术规格独立评审 | `pending` | 待评审 |
| 技术规格独立评审 | `approved` | 已通过 |
| 技术规格门禁、实施门禁 | `blocked` | 未开放 |
| 技术规格门禁、实施门禁 | `open` | 已开放 |

置信度保持原数字，不附加或改变单位。路径、主题及其它自由文本内容保持原值。

### 空值与默认路径

- `requirements_path`、`requirements_topic`、`requirements_scope` 或 `spec_path` 的 canonical 值为 `null` 且没有可靠默认值时，用户可见值显示“未确定”。
- `plan_path` 的 canonical 值为 `null` 时，用户可见值显示“尚未创建”，表达计划阶段尚未产出文档，而不是路径选择不确定。
- 当工作流已能依据既有契约可靠派生默认路径时，路径字段显示实际默认路径，不用“未确定”或“尚未创建”覆盖可用事实。

### 计划评审状态

计划评审状态依赖计划是否已创建，并按以下语境展示：

| canonical 事实 | 用户可见值 |
| --- | --- |
| 计划尚未创建 | 未开始 |
| 计划已创建，评审状态为 `not-approved` | 未通过 |
| 计划已创建，评审状态为 `approved` | 已通过 |
| 计划已创建，评审状态为 `unknown` | 未知 |

“未开始”不得用于掩盖已经存在但评审未通过的计划，“未通过”也不得暗示计划文档尚未创建。

### 单一权威展示

- 每个交接点只输出一个用户可见状态快照，字段顺序保持既有八字段或十四字段契约的顺序。
- 用户可见快照的字段名和可枚举状态值全部使用本 PRD 定义的中文，不在其后追加等价英文快照。
- 中文快照是 canonical 状态的展示视图，不成为新的机器权威来源，也不反向改写 canonical 字段和值。

### 内部 canonical 契约与兼容性

- `requirements_*`、`spec_*`、`plan_*`、`specification_gate` 和 `implementation_gate` 等现有字段继续作为内部权威机器字段。
- `pending`、`approved`、`unknown`、`blocked`、`open`、`not-approved` 等现有值继续用于内部校验、评估、序列化与工作流判断。
- 已有英文八字段和十四字段状态快照继续是有效输入；新输出的用户可见快照不要求使用者接触内部英文表示。
- 中文展示不得作为打开 canonical 门禁或推断批准状态的依据。只有既有权威状态满足条件时，门禁才可保持或变为 `open`。

### 映射失败的关闭行为

- 映射失败只指确定性实现缺陷，例如新增 canonical 枚举没有对应中文值、同一 canonical 状态存在相互冲突的中文映射，或渲染无法生成完整快照。
- 已定义的 canonical `unknown` 和本 PRD 定义的空值语境均属于正常输入，不构成映射失败。
- 映射失败时不得修改 canonical 快照、批准状态、独立评审状态或门禁状态。
- 映射失败时不得输出部分中文快照、中文与英文混合快照，或以旧英文用户可见快照静默降级。
- 映射失败只暂停依赖该状态快照的本次自动阶段交接或自动路由，不撤销已完成且经验证的上游事实。
- 系统必须报告可定位的确定性阻塞；修复后从失败时保留的原 canonical 快照重新映射和继续，不凭聊天记忆补造状态。

## 验收标准

1. 给定 PRD 工作流输出八字段交接，用户看到按既有顺序排列的八个中文字段名，所有可枚举状态值均为本 PRD 定义的中文值，且没有重复英文状态块。
2. 给定技术规格与计划工作流输出十四字段交接，用户看到按既有顺序排列的十四个中文字段名，所有可枚举状态值均为本 PRD 定义的中文值，且路径、主题和置信度原值完整保留。
3. 给定提示词生成工作流需要展示十四字段快照，用户可见状态同样遵循本 PRD 的中文映射，三个工作流 skill 不产生同一状态的不同译法。
4. 给定内部需求范围为 `product`、`phase` 或 `feature`，用户分别看到“产品”“阶段”“功能”，内部值保持不变。
5. 给定理解确认、用户批准或独立评审处于各自合法状态，用户看到与字段语境匹配的“待确认”“已确认”“待批准”“已批准”“待评审”“已通过”或“未知”。
6. 给定技术规格门禁或实施门禁为 `blocked` 或 `open`，用户分别看到“未开放”或“已开放”，中文展示不改变门禁的 canonical 值。
7. 给定需求、主题、范围或技术规格路径为空且没有可靠默认值，用户看到“未确定”；给定计划路径为空，用户看到“尚未创建”。
8. 给定存在可靠默认路径，即使输入路径为空，用户仍看到实际默认路径而不是空值提示。
9. 给定计划尚未创建、已创建但未通过、已通过或状态未知，用户分别看到“未开始”“未通过”“已通过”或“未知”。
10. 给定下游收到变更前产生的英文八字段或十四字段快照，工作流仍能读取、校验并保留其 canonical 语义。
11. 给定 canonical 状态为合法的 `unknown`，工作流正常显示“未知”并继续遵循既有门禁规则，不误判为映射失败。
12. 给定出现未覆盖枚举或冲突映射，工作流保留原 canonical 快照和全部门禁事实，停止本次自动交接，不输出部分中文、混合内容或英文降级块，并报告确定性阻塞。
13. 给定映射缺陷修复，工作流从失败时的原 canonical 快照重新生成完整中文快照，不从聊天记忆补造批准或门禁状态。
14. 给定任一受影响工作流生成机器使用的 YAML、JSON、脚本参数、评估结果或校验输入，其 canonical 字段名和值仍保持既有英文契约。

## 非功能需求

- 中文译法应自然、简洁，并能区分“待操作”“已通过”“未知”“尚未创建”和“门禁未开放”等不同产品语义。
- 同一 canonical 字段在不同 skill 中必须使用同一中文字段名；语义相同的状态不得出现无依据的多种译法。
- 中文展示不得增加第二份长期状态源，不得降低现有门禁、校验和错误可见性。
- 长路径、Unicode 文本与连续反引号等正文内容不得因状态中文化而被截断、转义错误或混入状态快照。

## 依赖、风险与假设

- 依赖现有工作流能够先产生并验证完整 canonical 状态，再生成用户可见视图。
- 假设既有八字段和十四字段的字段集合及顺序继续有效；若未来新增字段或枚举，必须同步定义中文展示后才能形成完整快照。
- 最大风险是中文展示被误当作新的机器契约，导致校验器或自动衔接从中文反向推断状态；本需求明确要求 canonical 英文状态始终为唯一权威来源。
- 上下文字段的空值含义不同，若一律翻译为同一个词会误导用户；本需求分别定义“未确定”和“尚未创建”。
- 中文映射覆盖不足会阻断一次自动交接，但保守阻断优先于输出不完整状态或错误打开门禁。
- 当前没有未解决的实质性产品问题。

## 技术规格交接约束

下游技术规格必须保留以下产品边界，并自行确定最小、可验证的实现方式：

- 三个受影响工作流 skill 的用户可见八字段和十四字段快照使用同一组中文字段名与语境化状态值。
- 内部 canonical 英文字段和值、字段顺序、批准与门禁语义、旧英文输入兼容性保持不变。
- 中文状态仅作为 canonical 快照的展示视图，不成为新的权威状态或反向解析来源。
- 映射失败必须保持 canonical 状态不变、关闭本次自动交接并显式报告，不得部分输出或静默回退英文。
- 技术设计必须覆盖正常状态、合法未知状态、上下文空值、计划跨字段状态、旧英文输入和确定性映射失败。
- 不修改真实 `CODEX_HOME`，不依赖本机安装副本或插件缓存路径实现 skill 间协作。
