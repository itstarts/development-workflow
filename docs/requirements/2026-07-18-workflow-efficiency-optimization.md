# 开发工作流效率优化

## 产品问题

当前五个 skill 已形成可靠的 PRD、技术规格、实施计划、开发提示词、受控实施和规则治理流程，但实际使用与维护仍存在可确认的重复成本：

- 普通澄清回复也持续展示完整八字段或十四字段状态，视觉噪声较大，用户需要反复跳过未变化的信息。
- 互不依赖的问题仍逐轮询问，增加了等待和往返次数。
- 文档型 skill 在进入具体阶段前即完整加载所有参考材料，占用不必要的上下文。
- 八字段和十四字段的用户可见映射、完整性判断和门禁保护分散在多个 skill 的自然语言规则中，维护者需要重复核对一致性。
- skill 运行时合同发生变化后，旧的 GREEN 输出和评审状态仍可能继续表现为当前有效证据，使维护者误判最新行为已经完成前向验证。
- 新会话开发提示词要求在每次委派前重复检查全局 Agent，目标会话内稳定不变的信息被反复读取。
- 仓库的定向测试、完整测试、官方 validator、plugin validator 和本地安装差异检查入口分散，维护者容易漏跑或重复执行。

这些问题不会直接破坏现有门禁，但会增加用户交互轮次、上下文成本、维护操作和证据复核成本，并降低对当前评估状态的信任度。

## 目标与成功指标

- 普通澄清回复只展示固定三行紧凑状态，使当前阶段、主题和下一步各自占一行，不重复阅读完整批准记录，也不把三个字段挤在同一长行。
- 暂停、确定性阻塞、确认或批准门、跨 skill 交接及会话路由继续展示完整八字段或十四字段，保持恢复与审计能力。
- 互不依赖的澄清问题可以在一轮内成组询问，依赖前一答案的问题继续逐个询问，不降低 95% 理解门。
- 每个 skill 只在当前阶段需要时加载对应参考内容，同时保留进入该阶段所需的全部规则和失败边界。
- 相同 canonical 状态产生一致、完整且可验证的 full 结果；compact 结果由同一 canonical topic 加同一已验证的当前阶段与下一步上下文唯一决定。两种展示逻辑都不得改变机器权威状态。
- skill 的当前运行时合同变化后，旧前向证据不能继续被认定为当前 `review-approved`；必须获得与当前合同匹配的新鲜评估和独立评审证据，或如实降级状态。
- 目标新会话中的全局 Agent 清单在首次需要委派时读取一次；没有失效证据时后续委派复用该清单。
- 维护者能够通过统一入口完成受影响 skill 的定向验证、仓库完整验证和只读安装差异检查，并获得明确的通过、失败和未运行结果。
- 效率优化不得改变已有 canonical 字段、批准状态、门禁计算、三态会话路由或旧英文输入兼容性。

## 非目标

- 本阶段不改变 `implementing-bounded-changes` 的独立评审完成门。
- 本阶段不实施轻量改动免评审。后续若单独推进，只允许不改变行为、配置语义、公共契约或文档内容的纯格式或机械型轻量改动免独立评审；任何文档改动仍必须独立评审。本 PRD 不授权该后续变化。
- 不合并现有五个 skill，不新增总路由 skill，也不改变它们的职责边界。
- 不改变八字段、十四字段、批准状态、门禁、会话路由和现有 development prompt renderer 的机器契约；本阶段只新增 handoff 状态的用户可见前向渲染合同。
- 不修改、迁移或操作 `agent-rules`、个人全局 Agent 配置或项目专属 Agent 角色。
- 不把用户可见中文状态变成新的机器状态源，不从紧凑状态或中文完整状态反向推断 canonical 状态。
- 不批量回写已明确归档的历史回复或历史评估；当前被声明为有效的 GREEN 与评审证据必须符合新鲜度要求。
- 不自动覆盖真实 `CODEX_HOME`，不自动安装、提交、push、merge、tag、release 或发布。
- 不在本 PRD 中确定脚本接口、文件组织、内部数据结构、并行实现或具体测试代码。

## 目标用户

- 使用中文与 Codex 进行需求澄清、技术设计和开发交接，需要快速识别当前进度与下一步的用户。
- 使用已批准 spec 和 plan 进入当前或新会话开发，需要稳定选择全局 Agent 的开发者。
- 维护本仓库 skill、评估证据、plugin 打包和本机安装状态的贡献者。

## 用户场景

### 场景一：普通需求澄清

用户正在回答预期内的产品问题，当前没有文件损坏、能力缺口或权限阻塞。回复正文包含本轮问题，末尾以连续三行分别显示当前阶段、稳定主题和当前需要完成的下一动作，不重复展示完整八字段。

### 场景二：普通技术澄清

已批准 PRD 进入技术规格阶段后，存在需要用户选择的正常技术方案。回复可以在同一轮询问互不依赖的问题，并以紧凑状态说明当前处于技术澄清及当前需要完成的下一动作；如果问题之间存在依赖，则只询问决定后续分支的当前问题。

### 场景三：到达门禁或需要恢复

工作流要求用户确认需求摘要、批准 PRD、批准 spec、批准继续实施，或用户要求暂停、跨会话恢复。回复展示完整八字段或十四字段，使用户能够复核当前文档、批准和门禁状态。

### 场景四：发生确定性阻塞

文件缺失或不可读、元数据冲突、评审者或下游能力不可用、权限冲突等事实阻止当前阶段继续。回复说明实际阻塞，并在状态映射可用时展示完整状态；若状态映射自身发生确定性缺陷，继续遵循现有无残缺状态块的关闭行为。

### 场景五：多个澄清问题可以并行回答

用户需要确认的多个问题互不改变彼此选项、含义或优先级。工作流在一轮内最多提出三个问题，并让用户一次回答；任何一个答案会改变后续问题时，不提前询问尚未稳定的问题。

### 场景六：维护一致的交接状态

PRD、spec/plan 和 session routing 工作流持有同一 canonical 事实。无论由哪个 skill 展示，同一 canonical 字段和值都得到一致 full 结果；当 canonical topic、已验证阶段和下一步上下文相同时，compact 结果也一致。完整状态、紧凑状态或显示失败均不改变批准与门禁事实。

### 场景七：维护评估证据

维护者修改 skill 的触发、运行时步骤、输出或失败合同。仓库不会继续把旧 GREEN 和旧评审结果显示为当前合同的有效完成证据；维护者必须补充新鲜前向结果和评审，或看到明确的未完成状态。

### 场景八：在新会话选择全局 Agent

目标新会话第一次需要委派时读取该会话实际加载的全局 Agent 清单。后续委派复用同一清单；只有配置已变化、按名称启动失败、清单读取失败或运行时证据冲突时才刷新。

### 场景九：运行仓库验证

维护者可以选择只验证一个受影响 skill，或执行完整仓库和 plugin 验证。统一入口报告实际执行的检查、结果和失败归因；只读安装检查能够比较候选与当前安装，但不修改安装目录。

## 范围与产品需求

### 回复类型与状态展示

- 本阶段改变用户可见状态行为的 skill 仅为 `creating-product-requirements`、`creating-development-specs-and-plans` 和 `generating-development-prompts`；另外两个 skill 的回复合同不在本阶段范围内。
- “普通澄清”是工作流预期内的发现或选择问题：当前输入没有损坏、冲突、权限失败、能力缺口或评审阻塞，也没有要求用户作出正式确认或批准决定。
- 普通澄清回复使用固定三行紧凑状态，格式和顺序固定为：

  ```text
  当前阶段：<阶段>
  主题：<topic、未确定或未知>
  下一步：<当前需要完成的动作>
  ```

  实际回复使用连续三行纯文本，不使用 Markdown code fence、列表标记、前导空格、字段分隔符或空白行，第三行后不再追加其它状态内容。
- 紧凑状态使用中文阶段和自然语言动作；稳定 topic、路径、命令和其它技术标识符保持原值。
- 紧凑状态中的 topic 只从 canonical 前向读取；当前阶段和下一步只从已经完成回复分类并验证过的当前回复上下文读取。三行结果不成为新的交接输入，不反向修改 canonical，也不替代工作流内部维护完整 canonical 快照。
- 以下情况必须展示完整八字段或十四字段：用户要求暂停或形成恢复点；发生确定性阻塞或能力缺口；请求用户确认需求摘要；请求用户批准 PRD、spec 或继续实施；完成一个文档阶段；选择下游 skill；执行 `current-session | new-session | blocked` 路由。
- 已有完整状态字段、顺序、中文映射、canonical 英文输入和门禁规则保持不变。
- 状态映射自身失败时继续 fail closed：保留 canonical 状态，不输出残缺、混合或推测状态，不继续自动交接，并报告可定位的确定性问题。

### 依赖感知的澄清批次

- 一轮最多询问三个互不依赖的实质性问题。
- “互不依赖”要求任一答案都不会改变其它问题的适用性、选项、含义、优先级或是否仍需询问。
- 当当前答案会决定后续问题分支时，只询问当前决定性问题，不提前制造选项。
- 批量询问不得使用默认值、行业惯例或推荐方案制造理解确定性；95% 理解置信度和用户确认门保持不变。
- 用户只回答部分问题时，保留已确认答案，仅继续询问仍然适用的缺失项。

### 按阶段加载参考内容

- skill 首次触发时只加载识别入口、当前阶段和安全边界所需的核心内容。
- 发现与澄清规则在进入澄清阶段前可用；文档合同在首次创建或实质修改文档前完整可用；评审和交接合同在首次评审、状态展示或阶段转换前完整可用。
- 不得以提高效率为由跳过当前阶段所需规则、降低门禁或依赖聊天记忆补造缺失内容。
- 当前阶段所需参考内容缺失、不可读或冲突时，工作流报告确定性阻塞，不静默继续或加载来源不明的替代内容。

### 一致且确定的状态结果

- canonical 英文字段和值继续是唯一机器权威。
- 对同一合法 canonical 快照，full 状态生成结果必须唯一、完整且可重复；三个受影响 skill 不得产生不同字段顺序、中文值或门禁显示。
- compact 状态的 topic 只读取 canonical；当前阶段与下一步读取已验证的当前回复上下文。同一 canonical topic、阶段和下一步输入必须产生相同三行结果；任何展示结果都不得打开、关闭或修改 canonical 门禁。
- 合法的 `unknown`、空路径语境、计划尚未创建和已有旧英文输入继续按现有产品含义处理。
- 新增或改变 canonical 枚举、字段或显示规则时，未覆盖映射必须在自动交接前被发现并关闭该次交接。

### 评估证据新鲜度

- 被声明为当前有效的 GREEN 和独立评审结果必须对应当前 skill 运行时合同，而不是较早版本的输出。
- 当运行时触发、步骤、输出、错误处理、交接或权限合同发生变化时，仓库必须阻止旧证据继续保持 `review-approved`，直到当前合同获得新鲜前向结果和独立评审。
- 纯粹不影响运行时合同的历史说明或归档材料不强制重跑；是否影响当前合同必须有可审查依据，不得默认为无影响。
- 验证结果必须能区分“当前证据有效”“实现已变化、证据待更新”和“证据或状态不可靠”，不得把后两者报告为完成。
- 新鲜度保护不得依赖用户手工记住需要更新哪些证据，也不得通过删除旧失败记录制造通过。

### 目标会话 Agent 清单复用

- 全局 Agent 清单以目标会话实际加载和可启动的能力为准，不以源仓库或个人安装目录的静态存在替代运行时事实。
- 目标会话第一次需要委派时读取并记录当前会话内清单，后续委派复用该清单。
- 仅在用户或运行时说明配置变化、按名称启动失败、初次读取失败、清单与实际能力冲突或用户明确要求刷新时重新检查。
- 复用清单不得把启动失败包装为成功，也不得在匹配 Agent 无法启动时静默改用不匹配角色。

### 统一验证与安装差异检查

- 维护者可以显式选择定向验证或完整验证；默认范围必须与变更影响匹配。
- 定向验证至少覆盖目标 skill 自身合同和与受影响公共交接直接相关的回归；完整验证覆盖仓库测试、五个 skill、仓库 validator、官方 skill validator 和 plugin validator。
- 可以并行执行彼此独立的检查，但最终结果必须逐项记录实际命令、退出状态和失败归因。
- 统一入口不得吞掉失败、把未执行检查报告为通过，或为了获得 GREEN 修改无关测试和行为。
- 安装差异检查默认只读，比较可发布 payload 与指定安装或 staging payload，忽略明确非发布的缓存与临时产物，并报告缺失、额外或内容不同的文件。
- 安装差异检查不授权覆盖、删除或修改真实安装；任何真实安装仍遵循单独授权和 staging 验证流程。

## 业务规则与产品可见错误

- 普通澄清与阻塞的分类以当前事实为准，不以回复篇幅、任务大小或 Agent 主观偏好决定。
- 用户正在作出的需求摘要确认属于正式确认门，不属于普通澄清，必须展示完整状态。
- 文档独立评审、用户批准和实施批准继续相互独立；紧凑显示不得隐藏一个当前需要用户决定的门。
- 如果无法可靠判断回复是否为普通澄清，保守展示完整状态，不得为了减少行数隐藏门禁或阻塞。
- 紧凑状态字段缺失、顺序错误、行数不是三行、topic 无法从 canonical 取得，或当前阶段/下一步上下文未经验证时，不输出猜测或残缺的三行状态；报告实际问题并遵循现有关闭行为。
- 评估新鲜度验证发现当前合同晚于当前 GREEN 或评审证据时，必须使验证失败或把状态降级为待完成，不得继续声明 `review-approved`。
- 统一验证入口中的任何必需检查失败时，整体结果不得显示为通过；环境问题应与实现失败分开报告。
- Agent 清单复用期间出现按名称启动失败时，当前委派停止并触发一次清单刷新；刷新后仍失败则报告能力缺口。
- 本阶段所有文档变更仍须按适用规则独立评审；后续格式或机械型轻量改动免评审规则不在本阶段生效。

## 验收标准

1. 给定 PRD 或技术规格阶段正在询问一个预期内、非阻塞的普通澄清问题，回复末尾只出现连续三行“当前阶段”“主题”“下一步”紧凑状态，顺序固定、行间无空行、没有字段分隔符，且不出现完整八字段或十四字段。
2. 给定用户要求暂停或形成恢复点，工作流请求需求摘要确认、PRD 批准、spec 批准或实施批准，完成一个文档阶段，或准备选择/已经选择下游 skill，回复展示当前阶段对应的完整八字段或十四字段。
3. 给定确定性阻塞、能力缺口或会话路由结果，回复展示完整状态；给定状态映射自身失败，回复保留 canonical 事实、停止自动交接且不输出残缺状态。
4. 给定三个互不依赖的澄清问题，工作流可以在一轮内同时询问；给定第四个独立问题，单轮仍不超过三个。
5. 给定后续问题的适用性或选项取决于当前答案，工作流只询问当前问题，不提前询问不稳定分支。
6. 给定批量问题只得到部分回答，工作流保留有效答案并继续询问仍适用的缺失问题，不重复已确认内容。
7. 给定 skill 处于发现阶段，未提前加载只在文档写入、评审或交接阶段需要的完整参考内容；进入对应阶段前，所需合同已经完整加载。
8. 给定当前阶段所需参考内容缺失或不可读，工作流报告阻塞且不越过当前门禁。
9. 给定相同合法 canonical 八字段或十四字段快照，所有受影响 skill 生成一致、完整的 full 中文显示；再给定相同的已验证当前阶段和下一步上下文，生成一致的 compact 三行显示。两种显示结果都不改变机器状态。
10. 给定合法 `unknown`、空值语境、计划未创建或旧英文输入，工作流保持现有兼容语义。
11. 给定 skill 运行时合同发生变化但 GREEN 与评审证据仍来自旧合同，仓库验证不允许该 skill 保持当前 `review-approved` 完成状态。
12. 给定当前合同获得新鲜前向结果和独立评审，评估状态可以恢复为 `review-approved`，并能指出对应当前合同的证据。
13. 给定目标新会话连续执行多次委派且没有清单失效证据，全局 Agent 清单只在首次委派前读取一次。
14. 给定 Agent 配置变化、按名称启动失败或清单与运行时冲突，工作流刷新清单并按最新事实继续或报告能力缺口。
15. 给定维护者选择一个 skill 的定向验证，统一入口只运行该 skill 与直接受影响的跨 skill 回归，并逐项报告结果。
16. 给定维护者选择完整验证，统一入口覆盖仓库测试、五个 skill 测试、仓库 validator、官方 skill validator 和 plugin validator，任何必需项失败都会使结果失败。
17. 给定多个检查互不依赖，统一入口可以并行执行并仍保留逐项命令、退出状态和失败归因。
18. 给定只读安装差异检查，可发布 payload 与指定安装内容一致时报告一致；存在缺失、额外或内容差异时准确列出，但不修改目标。
19. 给定本阶段的任意文档改动，适用的独立评审门保持有效，不因轻量改动的后续候选规则而跳过。
20. 给定完整变更完成，现有五个 skill 职责、canonical 英文合同、批准门、三态路由和旧输入兼容性均未发生未批准变化。

## 非功能需求

- 紧凑状态应保持固定三行、中文、可快速扫描，不重复完整状态中的批准字段，也不把三个字段压缩为单一长行。
- 状态生成、评估新鲜度和验证结果必须可重复、可审查，不依赖操作者记忆或不可见推断。
- 渐进加载不得以牺牲正确性、门禁真实性或失败可见性换取更少上下文。
- 统一验证入口不得增加新的运行时依赖，并应兼容仓库声明支持的 Python 版本。
- 所有输出继续避免回显密钥、令牌、凭证、个人隐私、真实 task/thread 标识符或本机私有数据。
- 效率优化应减少重复动作，不新增与实际风险不相称的安全、哈希、签名或评审门。

## 依赖、风险与假设

- 依赖三个交接 skill 继续维护完整 canonical 快照，即使普通澄清不再完整展示它。
- 三行紧凑状态减少了普通回复中的恢复信息，但比单行占用更多垂直空间；通过只保留三个必要字段，并在暂停、阻塞、确认、批准和交接点使用完整状态控制该取舍。
- 普通澄清与阻塞分类错误可能隐藏重要状态；保守回退到完整状态和前向场景用于控制该风险。
- 渐进加载可能遗漏后续阶段规则；每个阶段必须在首次相关动作前加载并验证所需合同。
- 评估新鲜度保护过宽可能导致无行为变化的维护也被迫重跑，过窄则会保留过期证据；下游技术设计必须定义可审查的行为影响边界。
- Agent 清单可能在目标会话期间失效；明确刷新条件用于避免每次扫描与永久缓存两个极端。
- 统一验证入口依赖现有测试与 validator 继续作为底层权威，不以新入口替代其真实结果。
- 当前没有未解决的实质性产品问题。

## 技术规格交接

下游技术规格必须保留以下产品边界，并自行确定最小、可验证的实现方式：

- 普通澄清使用固定三字段三行状态；暂停、阻塞、确认或批准门、跨 skill 交接和会话路由使用现有完整状态。
- canonical 英文字段和值、批准与门禁语义、旧英文输入兼容性和状态映射失败的关闭行为保持不变。
- 依赖感知的批量提问最多三个问题，不降低 95% 理解门或用户确认门。
- references 按阶段加载，但任何动作执行前必须具备该阶段完整适用规则。
- 同一 canonical 快照产生一致且可验证的 full 状态；同一 canonical topic 加同一已验证阶段/下一步上下文产生一致的 compact 三行状态；不从任何用户可见状态反向推断机器事实。
- 当前运行时合同变化后，旧 GREEN 与评审证据不得继续作为当前 `review-approved` 证据。
- 目标会话的全局 Agent 清单默认一次读取、条件刷新，启动失败必须如实处理。
- 统一验证入口支持定向、完整和只读安装差异检查，保留现有底层验证结果并且不修改真实安装。
- 本阶段不改变独立评审门；未来仅纯格式或机械型轻量改动可考虑免评审，任何文档改动仍需独立评审。
