# 工作流文档元数据中文化

## 产品问题

本工作流生成的 PRD、技术规格和实施计划已经使用中文标题、章节和解释性正文，但文档开头的 YAML frontmatter 仍使用英文键和英文状态值。中文用户在阅读文档时会遇到明显割裂，需要额外理解 `document_type`、`understanding_user_confirmation`、`independent_review` 等机器字段，才能判断文档类型和门禁状态。

这些元数据同时参与文档识别、批准校验和跨阶段衔接。直接替换而不保留兼容边界，可能使历史英文文档无法继续使用，或使中文显示结果反向改变既有批准和门禁事实。

## 目标与成功指标

- 本功能生效后新创建的 PRD、技术规格和实施计划在文档开头使用中文元数据字段名，并按字段语境显示自然、准确的中文状态值。
- 主题、路径、日期、置信度数字、评审角色标识等技术性内容保持原值，不因中文化失去可识别性或可复制性。
- 历史英文 frontmatter 文档继续可被现有工作流读取、校验和用于后续阶段，不要求用户手工迁移。
- 文档内的中文元数据只是既有产品状态的呈现和持久化形式，不改变批准、独立评审、技术规格门禁或实施门禁的业务含义。
- 同一语义字段缺失、重复、冲突或无法可靠识别时，工作流不选择有利值，不自动开放后续门禁。

当以下结果同时成立时，本功能视为成功：三个文档类型的新产物均使用约定的中文元数据；已批准的历史英文文档仍可进入原有后续流程；中文和英文表示在相同事实下得出相同门禁结论；歧义元数据不能获得批准状态或开放门禁。

## 非目标

- 不修改 Skill 在 Codex 界面中的展示名称，也不处理 Skill 名称是否显示为中文的问题。
- 不翻译八字段、十四字段 canonical 交接中的内部字段名、JSON 字段、脚本参数、命令行参数、API、协议或代码标识符。
- 不改变已经完成的用户可见八字段和十四字段中文状态快照功能。
- 不批量改写现有 PRD、技术规格、实施计划、历史评估输出或 Git 历史。
- 不把仓库中所有 YAML 文件或所有 frontmatter 一并中文化；范围仅包括本工作流生成和消费的 PRD、技术规格与实施计划。
- 不翻译文档路径、稳定主题、日期、命令、文件名、置信度数字以及 `product-analyst` 等评审角色标识符的实际值。
- 不在本 PRD 中决定解析器结构、映射表存放位置、文件修改清单、迁移脚本或测试实现。

## 目标用户

- 使用中文阅读和批准 PRD、技术规格与实施计划的工作流用户。
- 需要继续使用历史英文文档完成后续设计、计划或开发交接的用户。
- 维护文档生成、校验和门禁契约的工作流贡献者。

## 用户场景

### 场景一：阅读新生成的 PRD

用户完成需求理解确认后打开新 PRD。文档开头显示“文档类型”“主题”“范围类型”“需求理解确认”“用户批准”“独立评审”等中文字段，用户无需先理解英文机器字段即可判断当前状态；主题、日期和角色标识仍保持原始格式。

### 场景二：阅读新生成的技术规格和实施计划

用户查看技术规格或实施计划时，来源文档、来源批准状态、技术规格门禁和计划评审状态均使用一致的中文字段和中文状态语义。不同文档中的同一业务状态不会出现互相矛盾的中文表达。

### 场景三：继续使用历史英文文档

用户提供一个在本功能上线前生成、元数据完整且已批准的英文 frontmatter 文档。工作流继续按原有事实读取和校验，不要求复制内容、手工改键或重新批准。

### 场景四：处理技术性元数据值

用户查看中文 frontmatter 时，`codex-workflow-single-video`、仓库相对路径、ISO 日期、置信度数字和 `product-analyst` 等值保持原样，避免翻译后无法复制、匹配或追踪。

### 场景五：处理重复或冲突的双语元数据

同一语义字段同时出现中英文表示、重复声明、值不一致或无法确定对应关系时，工作流将受影响状态视为不可靠并保持后续门禁关闭。系统向用户指出元数据问题，不静默挑选更有利的批准值，也不把口头确认代替文档修复。

## 范围与产品需求

### 文档覆盖范围

- 本功能生效后从头创建的 PRD、技术规格和实施计划必须使用中文 frontmatter 字段名。
- frontmatter 继续位于文档开头，并保持单层、可明确识别的元数据结构。
- 新文档只写一套中文字段，不同时附加一套英文别名作为重复元数据。
- 功能生效前已存在的英文 frontmatter 文档继续保持英文格式；后续读取、维护、内容修改或重新评审均不触发隐式语言转换。只有用户另行明确要求迁移时，才进入不属于本功能的迁移范围。

### 生命周期中的完整元数据

| 文档与阶段 | 必须存在的中文字段 | 状态变化与新增字段 |
| --- | --- | --- |
| PRD 初次创建 | 文档类型、主题、范围类型、理解置信度、需求理解确认、用户批准、独立评审 | 用户批准为“待批准”，独立评审为“待评审” |

| PRD 用户批准 | PRD 独立评审通过的全部字段 | 用户批准改为“已批准”，新增批准日期 |
| 技术规格初次创建 | 文档类型、主题、需求文档、需求主题、需求范围、需求理解置信度、需求理解确认、需求文档用户批准、需求文档独立评审、技术规格门禁、技术规格用户批准、技术规格独立评审 | 来源状态固定为已确认、已批准、已通过且门禁已开放；当前规格用户批准为“待批准”，当前规格独立评审为“待评审” |
| 技术规格独立评审通过 | 技术规格初次创建的全部字段 | 技术规格独立评审改为“已通过”，新增技术规格独立评审角色、技术规格独立评审日期 |
| 技术规格用户批准 | 技术规格独立评审通过的全部字段 | 技术规格用户批准改为“已批准”，新增技术规格批准日期 |
| 实施计划初次创建 | 文档类型、主题、技术规格、技术规格用户批准、计划评审状态 | 技术规格用户批准固定为“已批准”，计划评审状态为“待评审” |
| 实施计划评审通过 | 实施计划初次创建的全部字段 | 计划评审状态改为“已通过”，新增计划评审角色、计划评审日期 |

“未通过”“未知”“未开放”等 canonical 或派生状态不作为本功能新增的有效文档持久化值。评审提出修改时，文档中的对应评审状态保持“待评审”，直至当前版本真实评审通过；缺失、冲突或无法识别的元数据由工作流对外判定为不可靠，不通过把“未知”写入文档来修补。

同一个内部事实在不同文档中必须得到一致且唯一的中文含义；路径、主题、日期、置信度数字和角色标识符保持实际值，不得因中文化改变。

### 兼容与门禁

- 工作流必须能识别符合既有契约的完整英文 frontmatter，以及符合本功能约定的完整中文 frontmatter。
- 英文和中文文档表达相同事实时，必须得到相同的批准、评审和门禁结果。
- 新生成的中文文档能够按原有阶段顺序继续进入技术规格、实施计划和开发交接，不要求用户额外转换格式。
- 中文化不得改变 stable topic、范围类型、置信度、理解确认、用户批准、独立评审或来源路径之间的既有一致性校验。
- 同一语义字段出现多次、中英文重复、值冲突、格式不受支持或无法关联到当前文档版本时，受影响字段必须视为未知或不可靠；依赖它的门禁保持关闭。
- 内部八字段和十四字段 canonical 交接继续使用既有英文字段和值。中文 frontmatter 不作为重新解释或放宽 canonical 门禁的依据。

## 业务规则与产品可见错误

- “新生成文档”只指本功能生效后从头创建的 PRD、技术规格或实施计划，不包括此前已经存在的英文文档。
- 历史英文文档无需仅为中文显示而重置批准、重新评审或修改时间；后续维护仍保持英文 frontmatter，发生产品含义变化时只按既有批准失效规则处理，不附带格式迁移。
- 文档元数据中文化只改变文档中的字段和值表示，不授予任何新的批准，不恢复已失效批准，也不把 `pending`、`unknown` 或缺失状态提升为已通过。
- 中英文语义映射必须唯一。无法唯一映射时，系统报告受影响元数据不可靠，并保持技术规格门禁或实施门禁关闭。
- 对重复或冲突字段，系统不得依据出现顺序、语言偏好或更有利的状态自动选择结果。
- 读取历史英文文档时，不应仅因其未中文化而显示错误或要求迁移。
- 新文档若未按约定完整生成中文元数据，系统不得把不完整结果描述为已完成中文化。

## 验收标准

1. 给定功能生效后用户新建 PRD，文档开头包含 PRD 映射表和生命周期矩阵规定的全部且仅有一套中文元数据；初次创建、独立评审通过、用户批准三个阶段分别使用规定字段和值。
2. 给定功能生效后用户新建技术规格，文档开头包含技术规格映射表和生命周期矩阵规定的全部且仅有一套中文元数据；来源 PRD 的路径、主题、范围、批准和评审事实清晰区分于当前技术规格的批准与评审事实。
3. 给定功能生效后用户新建实施计划，文档开头包含实施计划映射表和生命周期矩阵规定的全部且仅有一套中文元数据；初次创建和评审通过两个阶段分别使用规定字段和值。
4. 给定中文 frontmatter 中的路径、stable topic、ISO 日期、置信度数字或评审角色标识符，这些值保持原始格式且可继续用于匹配和追踪。
5. 给定一份元数据完整、状态可靠的历史英文 PRD，工作流继续按原有门禁进入技术规格流程，不要求迁移或重新批准。
6. 给定一份元数据完整、状态可靠的历史英文技术规格或实施计划，工作流继续按原有门禁完成后续阶段判断。
7. 给定分别使用英文和中文元数据、但表达相同事实的文档，两者得到相同的批准、评审和门禁结论。
8. 给定新生成的中文 PRD、技术规格和实施计划，工作流内部的八字段和十四字段 canonical 交接仍使用既有英文机器契约，且门禁语义不变。
9. 给定同一语义字段同时出现中英文键、重复声明或冲突值，工作流不选择其中的批准值，受影响状态被视为不可靠，依赖门禁保持关闭。
10. 给定字段缺失、中文状态不受支持或映射无法唯一确定，工作流报告可定位的元数据问题，不输出伪造的已批准或已开放状态，也不把“未知”“未通过”或“未开放”写入文档冒充有效生命周期状态。
11. 给定已有历史英文文档，启用本功能以及后续维护、内容修改或重新评审都不会隐式转换其 frontmatter、重置有效批准或产生仅为中文化的 Git diff。
12. 给定 Skill 展示名称、命令、API、JSON 字段或仓库中其它 YAML 文件，本功能不改变这些范围外内容。

## 非功能需求

- 中文字段和值应简洁、自然、语义一致，不通过中英文双写增加阅读负担。
- 中文和英文兼容输入的识别结果必须确定、可重复；相同输入不能因读取顺序产生不同门禁结果。
- 兼容历史文档时不得降低现有对缺失、重复、冲突和不受支持元数据的严格校验。
- 文档中文化不得回显或引入密钥、令牌、凭证、个人隐私或真实本机绝对路径。

## 依赖、风险与假设

- 依赖当前 PRD、技术规格、实施计划的文档类型、批准状态、评审状态和阶段门禁已有稳定语义。
- 依赖跨 Skill 的八字段和十四字段 canonical 英文契约继续作为机器交接真源。
- 最大风险是把用户可见中文值误当成新的业务状态，或在兼容旧文档时把重复、冲突状态错误提升为批准；本需求通过语义等价和歧义关闭门禁约束该风险。
- 中文字段可能与既有英文解析约定发生兼容张力；下游技术规格需要覆盖所有创建者和消费者，不得只修改模板显示。
- 假设用户要求的是实际写入 PRD、技术规格和实施计划 frontmatter 的中文字段和值，而不只是额外展示一段中文摘要。
- 当前没有未解决的实质性产品问题。

## 技术规格交接

下游技术规格必须保留以下产品约束，但由技术设计决定具体实现方式：

- 三种新文档均写入单一、完整的中文 frontmatter，不并列生成中英文两套元数据。
- 文档类型、范围、确认、批准、评审和门禁值按业务语境使用唯一中文含义；路径、主题、日期、数字和角色标识保持原值。
- 现有完整英文 frontmatter 继续作为兼容输入，不批量迁移，不因语言差异重置有效批准。
- 所有受影响的文档创建、读取、校验和阶段衔接路径必须得出一致结果，不能只改变模板外观。
- 同一语义字段重复、冲突、缺失或无法唯一映射时按不可靠状态处理，并保持依赖门禁关闭。
- 八字段和十四字段 canonical 交接继续使用英文机器契约；本功能不改变其字段、允许值或批准语义。
- Skill 展示名称、其它 YAML、脚本参数、JSON、命令、API 和历史文档批量迁移保持在范围外。
