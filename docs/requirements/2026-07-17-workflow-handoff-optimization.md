# 开发工作流交接体验优化

## 产品问题

当前开发工作流能够依次产出 PRD、技术规格、实施计划和新会话开发提示词，但交接体验仍存在以下问题：

- 面向用户的文档标题和正文包含不必要的英文，增加中文用户的理解成本。
- `creating-development-specs-and-plans` 完成全部门禁后，需要用户再次手动调用 `generating-development-prompts`，工作流衔接不连续。
- PRD 获得用户批准后，当前 `creating-product-requirements` 边界要求流程结束，用户即使已经表达继续推进意图，也需要再次发起技术规格流程。
- `generating-development-prompts` 默认生成新会话提示词，没有先判断当前会话是否更适合继续开发。
- 新会话提示词作为普通正文输出，需要用户手动选中，不能稳定一键复制。
- 全局 Agent 与本项目专属 Agent 的职责边界缺少统一判断，可能造成不必要的迁移、删除或重复维护。

这些问题使用户需要额外理解英文内容、手动选择下一步并整理提示词，也容易把“通用全局能力”和“项目专属评审能力”错误地视为二选一关系。

## 目标与成功衡量

- PRD、技术规格、实施计划及相关面向用户的说明默认使用中文，技术内容保留必要的英文形式。
- PRD 获得用户批准后，无需用户再次输入调用指令，在同一会话无条件进入技术规格流程。
- 技术规格与计划工作流完成全部批准和评审门禁后，无需用户再次输入调用指令，即可得到当前会话或新会话的开发建议。
- 实施门禁未打开时，保留用户显式请求带阻断门提示词的现有能力，不把自动衔接限制扩展为手动能力回退。
- 在自动衔接路径中，只有推荐新会话时才自动生成开发提示词；门禁未打开或阻塞状态下仍保留用户显式请求带阻断门提示词的能力。推荐当前会话时等待用户明确批准，不直接开始实施。
- 判断不明确时采用一致且保守的结果，默认推荐新会话。
- 已有明确证据证明当前会话和新会话都无法执行时，如实报告阻塞，不把新会话描述为解决方案。
- 新会话提示词能够作为一个完整单元一键复制，不需要用户手动框选或拼接。
- 全局 Agent 继续保持跨项目复用价值；本阶段逐个识别具备完整替代证据的项目级 Agent 删除候选，但不执行删除，没有证据证明完全覆盖时保持原状。

## 非目标

- 不在本阶段创建用于生成或维护项目专属 Agent 的新 skill。
- 不把项目专属 Agent 的创建能力加入 `managing-agents-rules`。
- 不迁移、修改或操作 `agent-rules` 仓库及其实际安装状态。
- 不要求每个项目都创建项目专属 Agent。
- 不在本阶段删除项目级 Agent 配置；本阶段只形成逐角色重叠分析和有证据的删除候选。
- 不在会话适用性判断后自动开始实施。
- 不改变 PRD、技术规格和计划已有的用户批准及独立评审门禁。
- 不允许两个 skill 通过读取兄弟 skill 安装目录、导入兄弟源码或固定本机路径形成运行时依赖。
- 不在本 PRD 中确定会话判断的具体评分算法、实现架构、脚本接口或文件修改方案。

## 目标用户

- 使用中文与 Codex 协作，通过本项目创建 PRD、技术规格、实施计划并进入开发阶段的用户。
- 需要在当前会话延续上下文与新会话隔离执行之间作出选择的开发者。
- 维护本项目 skill、交接契约和评审门禁的贡献者。

## 用户场景

### 场景一：生成中文需求与开发文档

用户通过工作流创建 PRD、技术规格或实施计划。最终文档的标题、章节和解释性正文使用中文；命令、API、字段、协议、文件名、路径、代码标识符、YAML 元数据键和必要状态值保持其技术形式，避免错误翻译。

### 场景二：PRD 批准后进入技术规格流程

PRD 已完成独立评审，用户明确批准当前文件。工作流先记录并验证批准状态，产生完整的 PRD 交接记录，然后无需用户再次输入 skill 名称或“继续推进”，在同一会话自动进入技术规格流程。

### 场景三：适合在当前会话继续开发

技术规格和计划已完成规定门禁。工作流自动结合当前会话上下文、任务规模、工作区状态、权限和可用能力进行判断，认为当前会话能够可靠继续。系统说明判断依据并建议继续，但在用户明确批准前不实施，也不生成新会话提示词。

### 场景四：推荐新会话开发

技术规格和计划已完成规定门禁，但当前会话的上下文负担、任务复杂度、预计持续时间、工作区状态或能力边界更适合隔离执行。系统说明推荐新会话的主要依据，并生成一份完整、可直接使用的新会话开发提示词。

### 场景五：无法明确判断会话适用性

可用证据不足以可靠判断当前会话能否承载后续开发。系统不把不确定性包装成当前会话可继续，而是默认推荐新会话并生成提示词，同时说明判断存在不确定性。

### 场景六：两个会话均存在确定性阻塞

已有明确证据证明当前会话和新会话都缺少必要权限、工具、文档或 Agent 能力。系统报告阻塞原因，不把新会话描述为能够解决问题，也不自动生成开发提示词。用户仍显式要求提示词时，可以获得一份保留明确阻断门的完整提示词。

### 场景七：复制新会话提示词

用户获得新会话提示词后，可以从一个独立区域一键复制全部内容。复制结果包含完整开发目标、文档路径、规则、权限、验证和评审要求，不需要拼接多个片段。

### 场景八：评估项目专属 Agent

维护者逐个比较本项目专属 Agent 与已存在的全局 Agent。依赖本项目 skill 契约、RED/GREEN 证据、plugin 打包或专属完成门禁的角色继续保留；只有在全局角色完整覆盖其职责、输入证据、只读边界和批准条件时，才可以认定为重复候选。

## 范围与产品需求

### 中文输出

- 面向用户生成的 PRD、技术规格和实施计划必须默认使用中文标题、中文章节名和中文解释性正文。
- 会话判断结论、推荐理由、阻塞说明和提示词中的自然语言必须默认使用中文。
- 技术标识符、命令、API、字段、协议、文件名、路径、代码片段、YAML 元数据键及必要状态值可以保留英文或其既有形式。
- 不得为了追求全中文而翻译会导致执行错误、契约变化或识别困难的技术内容。

### PRD 到技术规格的自动衔接

- 当当前 PRD 的理解确认、独立评审和用户批准全部有效，且 `specification_gate` 真实为 `open` 时，工作流必须在同一会话自动进入 `creating-development-specs-and-plans`。
- 自动衔接是每次 PRD 批准后的固定行为，不以用户额外输入“继续推进”、再次点名 skill 或重复提供 PRD 路径为前提。
- 进入下一阶段前必须先完成 PRD 批准元数据写入、写后验证及完整八字段交接，不得用自动衔接跳过 PRD 自身完成条件。
- PRD 未经用户批准、独立评审未通过、理解确认失效、元数据不可靠或 `specification_gate` 未打开时，不得进入技术规格流程。
- 当前运行时无法继续执行技术规格流程时，必须说明能力缺口并保留真实门禁状态，不得声称已经进入下一阶段。
- 自动衔接只表示主 Agent 按顺序切换两个独立工作流，不允许任一 skill 读取兄弟 skill 安装目录、导入兄弟源码或依赖固定本机安装路径。

### 工作流自动衔接

- 当 `creating-development-specs-and-plans` 完成当前主题的全部批准和评审门禁、达到可进入实施阶段的状态后，应在同一会话自动进入会话适用性判断。
- 用户不需要再次输入 skill 名称、复制文档路径或重复描述开发目标。
- 进入会话适用性判断前，必须先完成并验证 requirements、spec、plan 与双门状态的完整十四字段交接，不得用自动衔接省略或延后现有交接记录。
- 实施门禁尚未打开时，不自动进入会话适用性判断。用户仍可显式请求新会话提示词，但提示词必须保留真实实施阻断门，不得把生成提示词表述为实施门禁已经打开。

### 会话适用性判断

- 判断必须使用当前可获得的真实证据，不得凭单一固定因素或无证据猜测得出结论。
- 判断至少覆盖以下维度：
  - 当前会话是否具备完整、一致且仍然相关的项目上下文；
  - 当前会话是否存在无关内容、相互冲突的要求或明显上下文负担；
  - 后续实施的范围、复杂度、风险和预计持续时间；
  - 当前工作区、分支、批准状态和验证条件；
  - 实施所需工具、权限和 Agent 能力在当前会话是否可用；
  - 继续使用当前会话是否存在明显的信息丢失、范围漂移或执行偏差风险。
- 判断结果只能是“建议当前会话继续”“推荐新会话”或“阻塞”。证据不足、维度冲突或无法可靠判断时，归入“推荐新会话”；已有明确证据证明两个会话都无法执行时，归入“阻塞”。
- 输出必须说明影响结论的主要依据，使用户能够理解和复核建议。
- 不得把确定性阻塞降级为判断不明确，也不得用推荐新会话掩盖新会话同样缺少必要能力的事实。

### 当前会话继续

- 建议当前会话继续时，不生成新会话开发提示词。
- 系统必须等待用户对继续开发作出明确批准；此前不得开始修改代码、配置或其它实施文件。
- 用户拒绝或改为选择新会话时，应按新会话路径生成提示词，不把此前建议视为实施授权。

### 新会话提示词

- 结论为“推荐新会话”、用户主动选择新会话，或用户在门禁未打开及阻塞状态下显式请求时，可以生成开发提示词。
- 门禁未打开或存在确定性阻塞时生成的提示词，必须明确保留对应阻断门，不得授权目标会话开始实施。
- 提示词必须保持完整、自包含，并保留已有的文档路径、评审状态、仓库证据、规则、权限、验证和独立评审要求。
- 提示词必须放入一个独立、完整、支持一键复制的区域，例如具备复制操作的 Markdown 代码框。
- 不得把需要复制的提示词拆成多个代码框、正文片段或需要用户重新整理的部分。
- 复制区域之外可以简要说明推荐理由，但不得把执行所需内容留在复制区域之外。

### 全局与项目专属 Agent 边界

- `agent-rules` 继续作为通用全局 Agent 的维护边界，本阶段不把其内容迁移到本项目。
- 全局 Agent 用于跨项目可复用的通用职责，为没有项目专属配置的项目提供基础能力。
- 本项目可以保留依赖本仓库特有 skill 契约、评估证据、plugin 打包和专属门禁的项目级角色。
- 项目级角色是否可被替代，以本项目对所有维护者能够稳定保证的 Agent 能力为权威依据；`agent-rules` 的版本化配置和个人实际安装状态只能作为只读辅助证据。
- 单个用户的个人环境已安装某个全局角色，不能证明该角色对本项目全部维护者稳定可用，也不能单独构成删除项目级角色的依据。
- 每个项目级角色必须单独评估，不得因为存在全局 Agent 而整体删除项目级配置。
- 只有当等价角色对本项目全部维护者具有稳定、明确且可验证的可用性，并完整覆盖项目级角色的职责范围、必要输入、输出结论、只读和外部状态边界、批准条件及验证要求时，才能把该项目级角色认定为删除候选。
- 如果无法证明完整覆盖，默认保留项目级角色并记录尚未消除的职责差异。
- 本阶段的终点是逐角色评估及删除候选清单，不执行项目级 Agent 配置删除；后续删除需要单独批准的实施范围和届时仍有效的覆盖证据。

## 业务规则与用户可见错误

- “完成技术规格与计划工作流”是指当前主题达到已有契约规定的可实施状态，不得把文档已创建、评审待定或用户尚未批准视为完成。
- “PRD 批准后自动衔接”只在当前文件的理解确认、独立评审、用户批准和元数据验证全部成立时触发；旧版本批准或口头继续意图不能替代当前文件批准。
- 用户批准 PRD 后，即使没有附加“继续推进”等措辞，也应进入技术规格流程；不得因为 PRD skill 已完成而要求用户重复授权同一阶段转换。
- 自动衔接失败时必须报告实际失败点，不得把技术规格路径、评审状态或下一阶段启动情况补写为成功。
- 技术规格与计划完成后的十四字段交接尚未完整产生或验证时，不得进入会话适用性判断，也不得伪造缺失字段。
- 实施门禁未打开时，用户显式请求生成提示词不等于门禁打开；生成结果必须继续阻止实施。
- 当前会话建议不是实施批准；只有用户随后明确同意继续开发，才可以进入实施流程。
- 推荐新会话不代表系统自动创建、打开或操作用户可见的 Codex 会话，只提供可复制的提示词。
- 判断所需的关键状态缺失、冲突或不可读取时，必须说明不确定性并推荐新会话，不得静默假定当前会话可继续。
- 已知当前会话和新会话都无法满足必要条件时，必须报告阻塞，不得以“无法判断”为由生成正常实施提示词；用户显式请求时只能生成带阻断门的提示词。
- 新会话提示词无法完整生成时，必须报告阻塞原因，不得输出残缺或需要用户自行补全的提示词。
- 一键复制区域无法承载完整提示词时，不得通过删减必要执行约束来适配展示形式。
- 全局角色存在同名或近似职责不等于完整覆盖；无法核实职责和门禁时不得删除项目级角色。
- Agent 重叠评估没有发现可证明的重复角色时，允许本阶段不删除任何项目级配置。
- 即使形成删除候选，本阶段也不删除项目级 Agent 配置。

## 验收标准

1. 给定用户创建新的 PRD、技术规格或实施计划，最终文档的标题、章节和解释性正文使用中文，技术标识符和机器可读元数据保持正确形式。
2. 给定当前 PRD 已完成独立评审且用户批准当前文件，工作流写入并验证批准状态、输出完整八字段交接，然后无需额外用户指令即在同一会话进入技术规格流程。
3. 给定用户只回复批准而没有附加继续指令，工作流仍自动进入技术规格流程。
4. 给定 PRD 的当前独立评审、用户批准、理解确认或元数据验证任一不成立，工作流不自动进入技术规格流程。
5. 给定 PRD 已批准但当前运行时无法进入技术规格流程，工作流如实报告能力缺口和门禁状态，不声称下一阶段已启动。
6. 给定技术规格或计划尚未完成批准和评审门禁，工作流不自动进入会话适用性判断，也不生成可用于绕过门禁的开发提示词。
7. 给定实施门禁尚未打开但用户显式请求新会话提示词，工作流仍能生成完整提示词，并在其中保留明确实施阻断门。
8. 给定技术规格和计划已完成全部门禁，工作流先完成并验证 requirements、spec、plan 与双门状态的完整十四字段交接，再无需用户输入调用命令，在同一会话给出会话适用性判断。
9. 给定当前会话上下文完整、实施范围可控且所需能力可用，工作流说明主要依据并建议当前会话继续，同时等待用户明确批准且不生成新会话提示词。
10. 给定当前会话上下文负担明显、实施规模或风险更适合隔离，工作流说明主要依据并推荐新会话。
11. 给定不同判断维度相互冲突或证据不足，工作流默认推荐新会话并说明不确定性。
12. 给定明确证据证明当前会话和新会话都缺少必要能力，工作流报告阻塞，不把新会话描述为解决方案，也不自动生成提示词。
13. 给定确定性阻塞后用户仍显式请求提示词，工作流生成带明确阻断门的完整提示词，不授权目标会话实施。
14. 给定建议当前会话继续但用户尚未批准，工作流不开始任何实施修改。
15. 给定结论为推荐新会话，工作流生成一份内容完整、自包含的新会话开发提示词。
16. 给定生成的新会话提示词，用户能够从一个独立区域一键复制全部执行内容，不需要手动框选、拼接或补充区域外信息。
17. 给定结论为建议当前会话继续，工作流不生成无用的新会话提示词。
18. 给定用户在当前会话建议后改为选择新会话，工作流能够转为生成完整可复制提示词，并且不把此前建议视为实施授权。
19. 给定全局 Agent 仍由 `agent-rules` 维护，本阶段不迁移、修改或操作该仓库及其实际安装状态。
20. 给定本项目任一项目级 Agent，维护者能够基于稳定可用性、职责、输入、输出、边界、批准条件和验证要求形成单独评估。
21. 给定某个等价全局角色只存在于个人安装环境，工作流不会据此把项目级角色认定为删除候选。
22. 给定某项目级角色没有证据证明被一个对全部维护者稳定可用的角色完整覆盖，该角色保持不变。
23. 给定评估形成了删除候选，本阶段只报告候选及证据，不删除项目级 Agent 配置。
24. 给定没有任何项目级角色满足完整覆盖条件，本阶段保留全部项目级角色并如实报告评估结果。

## 非功能需求

- 中文输出应保持简洁、准确和自然，不以中英文重复标题增加阅读负担。
- 会话建议必须可解释，用户能够从输出中识别影响判断的主要事实。
- 新会话提示词的复制体验应适用于 Codex 当前支持的一键复制展示能力；不得依赖用户安装额外工具。
- 工作流不得回显密钥、令牌、凭证、个人隐私或其它不应进入文档和提示词的敏感信息。
- 变更后必须保持五个现有 skill 的职责独立，以及现有文档批准、评审和实施门禁的真实性。

## 依赖、风险与假设

- 依赖当前会话能够获得技术规格、计划、仓库状态、适用规则和必要的上下文证据。
- 依赖 Codex 客户端能够为某种独立展示区域提供一键复制能力；Markdown 代码框是当前可接受的表现形式。
- 假设 YAML 元数据键、状态值、skill 名称、命令和代码标识符属于可以保留英文的技术内容。
- 会话适用性判断具有上下文相关性，同一计划在不同会话状态下可能得到不同建议；可解释依据和不确定时默认新会话用于控制该风险。
- 确定性阻塞与判断不明确具有不同产品含义；前者不应通过推荐新会话被掩盖。
- 自动衔接可能与现有 skill 自包含和职责独立约束产生张力；下游技术规格必须在不绕过门禁、不依赖本机安装路径的前提下解决该交接。
- PRD 批准后的自动阶段切换依赖当前运行时能够继续选择技术规格工作流；能力不可用时必须暴露限制，不能用伪造交接状态兜底。
- 项目级和全局 Agent 可能存在部分职责重叠，但部分重叠不足以授权删除；误删专属角色会削弱项目评审质量。
- 当前没有未解决的实质性产品问题。

## 技术规格交接约束

下游技术规格必须保留以下产品约束，但由技术设计决定具体实现方式：

- 所有面向用户的自然语言文档与说明默认中文，技术内容保留正确的原始形式。
- 当前 PRD 全部门禁打开后，主 Agent 必须在完整记录和验证 PRD 交接状态后，无条件在同一会话进入技术规格流程；两个 skill 仍保持源码和安装边界独立。
- 只有在当前主题的技术规格、计划及既有批准和评审门禁全部满足后，才自动进入会话适用性判断。
- 进入会话适用性判断前必须完成并验证现有十四字段交接，自动衔接不得削弱 requirements、spec、plan 与双门状态记录。
- 实施门禁未打开时保留用户显式生成带阻断门提示词的能力，但不触发自动会话判断。
- 判断必须综合多维真实证据并输出可解释依据；不能明确判断时默认推荐新会话，两个会话均确定无法执行时报告阻塞。
- 建议当前会话继续时必须等待新的用户实施批准，不生成提示词，也不直接开发。
- 推荐新会话时只生成一份完整、自包含、可一键复制的提示词，不自动创建或操作会话；门禁未打开或阻塞状态下的显式提示词必须保留对应阻断门。
- 全局 Agent 继续由 `agent-rules` 维护；本阶段不迁移或操作该仓库。
- 项目级 Agent 逐个进行稳定可用性和职责等价性评估，只有面向全部维护者的可用性及完整覆盖得到证明时才允许作为删除候选；本阶段不执行删除。
- 不新增项目专属 Agent 创建 skill，也不扩张 `managing-agents-rules` 的职责。
