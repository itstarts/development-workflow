# AGENTS 规则治理

## Product Problem

新项目或现有项目可能缺少仓库根目录的 `AGENTS.md`，导致 Codex 在开始开发时无法及时获得项目级约束。开发过程中形成的稳定经验也可能只停留在当前会话，后续工作需要重复确认，或者继续遗漏已经验证过的项目规则和通用规则。

现有流程需要一种受用户控制、低打扰的规则治理能力：在真正开始修改项目前发现项目级规则缺失，并在开发完成后仅基于实际证据提出值得沉淀的规则候选。任何项目级或全局规则写入都必须由用户逐次批准。

## Goals and Success Measures

- 在每个项目的当前会话首次进入实质性开发任务时，能够在编辑文件前完成一次项目级 `AGENTS.md` 检查。
- 缺少项目级规则时只提示一次；用户拒绝后，本次会话不再重复提示项目级规则。
- 非 Git 项目能够获得一次初始化 Git 仓库的建议，同时不因拒绝 Git 初始化而阻塞规则创建或后续开发。
- 开发完成后，只在存在有证据、可复用且非临时的规则候选时集中提示一次；没有合格候选时保持零提示。
- 每次创建或更新项目级、全局 `AGENTS.md` 前都提供候选内容、归类理由和当前 diff，并取得当次明确批准。
- 已批准的修改保持无关既有内容和用户已有变更；无法安全写入时不给出成功声明。

## Non-goals

- 不自动或静默创建、修改任何 `AGENTS.md`。
- 不读取、修改、同步或操作 `agent-rules` 仓库；相关同步由用户自行完成。
- 不把一次授权延伸为本会话后续授权或长期授权。
- 不把当前任务的临时要求、未经验证的推断或仅适用于一次操作的细节持久化为规则。
- 不因缺少 Git 仓库或项目级 `AGENTS.md` 阻塞用户已经批准的开发工作。
- 不在只读分析、代码解释、评审、状态查询、日志检查、创建分支或纯 Git 操作中触发首次检查。
- 不在本功能中规定具体实现架构、文件扫描程序、状态存储方式或交互界面。

## Target Users

- 使用 Codex 创建新项目、且项目尚未初始化 Git 或尚未建立项目级规则的开发者。
- 使用 Codex 维护现有项目、希望把经过开发验证的项目约束及时沉淀到仓库规则中的开发者。
- 希望由 Codex 识别跨项目通用规则候选、但仍保留逐次审批权的开发者。

## User Scenarios

### 场景一：Git 项目缺少根目录规则

用户首次要求 Codex 在当前会话中实现功能、修复 Bug、重构，或者修改会影响项目行为的测试、配置或工程文档。Codex 在实际编辑前识别 Git 仓库根目录并检查根目录 `AGENTS.md`。文件缺失时，Codex 询问用户是否创建；用户拒绝后继续原任务，本次会话不再提示项目级规则。

### 场景二：非 Git 新项目

用户首次要求 Codex 在当前工作区执行实质性开发。Codex 以当前工作区根目录作为项目根目录，检查项目级 `AGENTS.md`，并提示用户是否初始化 Git 仓库。只有用户当次明确批准后才初始化；拒绝后本次会话不再提示 Git 初始化，但用户仍可独立选择创建项目级规则。

### 场景三：存在局部规则但缺少项目根规则

项目子目录存在 `AGENTS.md`，仓库或工作区根目录没有 `AGENTS.md`。Codex 将子目录文件视为局部规则，而不是项目级规则的替代，因此仍提示用户是否创建根目录规则。

### 场景四：开发完成后出现项目规则候选

本次开发验证了稳定的构建命令、测试命令、项目结构约束、业务边界或其它可重复使用的项目事实。Codex 在交付阶段集中展示项目级候选、归类理由和 diff，并询问是否更新。用户拒绝则不写入。

### 场景五：开发完成后出现全局规则候选

本次开发形成了能够跨项目稳定复用的协作、工程或验证原则。Codex 在交付阶段集中展示全局候选、归类理由和 diff，并询问是否直接更新当前生效的全局 `AGENTS.md`。只有本次明确批准后才写入，不操作 `agent-rules`。

### 场景六：用户此前拒绝项目级规则

用户在开发前已经拒绝创建项目级 `AGENTS.md`。开发完成后，即使出现项目规则候选，本次会话也不再提示项目级规则；如果存在独立且合格的全局规则候选，仍可单独提示一次。没有全局候选时完全不提示。

## Scope and Product Requirements

### 首次检查

- 每个项目在当前会话首次进入实质性开发任务时检查一次，并在实际编辑文件前完成检查。
- 实质性开发任务包括实现功能、修复 Bug、重构，以及修改测试、配置或会影响项目行为的工程文档。
- 只读分析、代码解释、评审、状态查询、日志检查、创建分支和纯 Git 操作不触发检查。
- Git 项目以 Git 仓库根目录为项目根；尚未初始化 Git 的新项目以当前打开的工作区根目录为项目根。
- 检查目标是项目根目录的 `AGENTS.md`。子目录中的 `AGENTS.md` 只表示局部规则，不能满足项目级规则检查。
- 项目级 `AGENTS.md` 已存在时，首次检查不产生缺失提示；完成阶段仍可按候选规则标准提出更新建议。
- 项目级 `AGENTS.md` 存在但无法读取时，不得将其视为缺失或建议覆盖。Codex 必须说明读取失败，在文件可读取前暂停实质性开发的文件编辑，但可以继续安全的只读诊断。

### 缺失提示和会话降噪

- 项目级 `AGENTS.md` 缺失时，先展示建议内容、归类理由和以不存在文件为基线的创建 diff，再询问用户是否批准创建。
- 用户拒绝创建后，当前会话不再提示创建或更新项目级 `AGENTS.md`，且不阻塞原开发任务。
- 当前工作区不是 Git 仓库时，建议用户初始化 Git。只有当次明确批准后才执行；拒绝后当前会话不再重复提示。
- Git 初始化和项目级规则创建是两个独立选择，任一拒绝都不自动否决另一项。

### 开发完成后的规则候选

- 开发完成后进行一次规则沉淀检查。
- 候选规则必须同时满足：有本次开发证据支持、能够在后续工作中重复使用、不属于当前任务的临时细节，并能够有效提升后续协作或开发效率。
- 有合格候选时，在交付阶段按目标文件集中提示一次；没有合格候选时不产生规则提示。
- 项目技术栈、构建和验证命令、架构边界、业务约束等项目事实归入项目级 `AGENTS.md`。
- 能跨项目稳定复用的协作、工程和验证原则可以建议归入全局 `AGENTS.md`。
- 提示必须说明每条候选的证据和归类理由，让用户能够判断其稳定性和适用范围。

### 逐次审批和写入

- 每次创建或更新前，展示目标文件、候选规则、归类理由和基于当前文件内容生成的 diff。
- 项目级和全局规则的每次写入都需要用户对本次具体修改明确批准。
- 一次批准只覆盖当次展示的 diff；内容发生变化或需要追加另一批规则时必须重新展示并再次批准。
- 全局规则获得批准后，允许直接更新 Codex 当前生效的全局 `AGENTS.md`，但不得读取、修改或操作 `agent-rules`。
- 写入必须保留无关既有内容和用户已有变更，不得用整文件替换掩盖未展示的修改。
- 写入后应向用户说明实际修改的目标和结果；未经验证不得声称规则已经更新或生效。

## Business Rules and Product-visible Errors

- “当前会话首次”按项目分别计算；同一会话进入另一个项目时，该项目仍有自己的一次首次检查。
- 用户拒绝项目级规则后，本次会话内的项目级提示保持关闭；这不阻止对独立全局候选进行一次提示。
- 用户拒绝 Git 初始化后，本次会话内的 Git 初始化提示保持关闭。
- 无证据、仅推测有用、与现有规则重复或只服务当前任务的内容不得作为候选提示。
- 不能可靠识别项目根目录时，必须说明原因并停止规则创建或更新，不得猜测目标路径；原只读工作可以继续，进入修改前由用户决定如何处理。
- 项目级 `AGENTS.md` 存在但因权限、编码或 I/O 错误无法读取时，必须说明错误，不得将文件视为缺失、不得覆盖，也不得进入实质性开发的文件编辑；修复可读性前只允许继续安全的只读诊断。
- 无法定位或读取当前生效的全局 `AGENTS.md` 时，必须说明未能确定目标，不得改写其它猜测路径。
- 目标文件在首次读取后、展示 diff 前或获得批准后变为不可读时，当前 diff 和批准均不再有效；必须按读取错误处理，恢复可读后基于最新内容重新展示 diff 并请求批准。
- 在展示 diff 后发现目标文件内容发生变化时，原批准失效；必须基于最新内容重新生成 diff 并再次请求批准。
- 写入因权限、文件冲突或其它错误失败时，必须明确报告未更新及错误影响，不得把建议或部分操作表述为完成。
- Git 初始化失败时，必须报告失败；该失败不自动阻塞项目级规则创建或用户原开发任务。

## Acceptance Criteria

1. 给定一个根目录没有 `AGENTS.md` 的 Git 项目，当当前会话首次进入实质性开发任务时，Codex 在编辑前正确使用 Git 根目录，展示建议内容、归类理由和创建 diff，并仅在用户批准该 diff 后创建文件。
2. 给定一个非 Git 新项目，当首次进入实质性开发任务时，Codex 以工作区根目录检查规则并建议初始化 Git；未经当次批准不执行初始化。
3. 给定根目录缺少规则但子目录存在 `AGENTS.md`，Codex 仍识别项目级规则缺失，同时保留局部规则的适用性。
4. 给定根目录存在可读取的 `AGENTS.md`，Codex 不产生缺失或创建提示；如果开发完成后也没有合格候选，则整个任务不产生规则治理提示。
5. 给定根目录存在但无法读取的 `AGENTS.md`，Codex 明确报告读取错误，不把它视为缺失、不覆盖它，并在恢复可读前只继续安全的只读诊断，不进入文件编辑。
6. 给定用户拒绝项目级规则，Codex 继续已批准的开发任务，且本次会话不再出现项目级创建或更新提示。
7. 给定用户拒绝 Git 初始化，Codex 本次会话不再重复提示 Git 初始化，并允许用户独立决定是否创建项目级规则。
8. 给定只读分析、解释、评审、状态查询、日志检查、创建分支或纯 Git 操作，Codex 不执行项目级规则缺失提示。
9. 给定同一会话先后进入两个项目，Codex 分别维护首次检查状态，第二个项目不会沿用第一个项目的已检查或已拒绝状态。
10. 给定开发完成后存在符合全部候选标准的项目规则，Codex 在交付阶段集中展示证据、归类理由和项目级 diff；未经明确批准不写入。
11. 给定开发完成后不存在合格候选，Codex 不输出规则更新提示。
12. 给定用户此前拒绝项目级规则但存在合格全局候选，Codex 只提示一次全局候选，不再次提示项目级规则。
13. 给定任意全局候选，Codex 在每次写入前展示当前 diff 并取得当次明确批准；批准后直接更新当前生效的全局 `AGENTS.md`，且不操作 `agent-rules`。
14. 给定用户已经批准某个 diff，但目标文件在写入前发生变化或变为不可读，Codex 不沿用原批准；恢复可读后基于最新内容展示 diff 并重新请求批准。
15. 给定创建或更新成功，结果保留目标文件中无关既有内容和用户已有变更，并提供实际写入验证；失败时明确报告未更新。

## Non-functional Requirements

- 提示应简洁并说明触发原因，避免在同一会话重复打扰用户。
- 候选规则和 diff 必须足以让用户在批准前理解写入内容、目标位置和适用范围。
- 规则内容不得回显或持久化密钥、令牌、凭证、个人隐私或其它不应进入规则文件的敏感信息。
- 行为应适用于 Git 项目和尚未初始化 Git 的新项目，并遵守当前环境已有的文件权限与安全边界。

## Dependencies, Risks, and Assumptions

- 依赖 Codex 能够识别当前工作区、Git 仓库根目录、当前会话内已提示和已拒绝的状态，以及当前生效的全局 `AGENTS.md`。
- 假设用户希望由 Codex 直接更新当前生效的全局规则文件，并自行在 `agent-rules` 项目中完成后续同步。
- 主要风险是把临时经验误判为长期规则、重复提示、错误归类到全局范围，或者基于过期文件内容写入；候选标准、会话降噪、归类理由、diff 和逐次审批用于约束这些风险。
- 若运行环境不能可靠保存当前会话内的提示状态，应优先保证不重复写入和不绕过审批，并把可能重复提示的限制明确告知用户。
- 当前没有未解决的实质性产品问题。

## Technical Specification Handoff

下游技术规格必须保留以下产品约束：

- 检查发生在当前会话首次实质性开发任务的实际编辑之前，并按项目隔离状态。
- Git 根目录、非 Git 工作区根目录和局部 `AGENTS.md` 的产品含义不能混淆。
- 存在但不可读取的项目级 `AGENTS.md` 不能按缺失处理或覆盖，恢复可读前不得进入实质性开发的文件编辑。
- 项目级规则、Git 初始化和全局规则分别授权，拒绝状态和降噪行为按本 PRD 执行。
- 完成阶段只有在候选规则同时满足证据、复用性、非临时性和效率价值时才提示；无候选必须零提示。
- 每次写入必须基于当前内容展示具体 diff 并取得当次批准，文件变化会使旧批准失效。
- 全局规则直接更新当前生效的全局 `AGENTS.md`，不触达 `agent-rules`。
- 所有写入必须保留无关内容、保护用户已有变更、避免敏感信息，并对成功或失败提供可核验结果。
