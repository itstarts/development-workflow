# Creating Development Specs and Plans

## Overview

Turn approved product requirements into repository-grounded technical artifacts through the preserved `standard` or `full` route. The standard route reviews spec and plan as one technical package; the full route preserves sequential spec approval and plan review. Preserve uncertainty and stop at gates instead of upgrading missing evidence.

## Workflow

1. First read only [references/discovery-and-clarification.md](references/discovery-and-clarification.md), then inspect applicable rules from the repository root to the working directory and relevant repository evidence. Never recursively search above the repository root. Before the first spec or plan write, read [references/document-contracts.md](references/document-contracts.md) completely. Before a blocked reply, review, approval, full-status reply, or routing transition, read [references/review-and-handoff.md](references/review-and-handoff.md) completely. Read each required reference once per current workflow unless it changes or the earlier read was unreliable; a missing, unreadable, or conflicting required reference stops that stage.
2. Accept the explicit eight-field handoff from the approved upstream workflow or require an explicit product requirements path plus an expected topic and expected scope from the user. Use its `requirements_path`, `requirements_topic`, and `requirements_scope` as independent inputs. Do not derive expected identity from the PRD itself, its filename, or a guessed default. Run `scripts/inspect_product_requirements.py` with the repository root, requirements path, expected topic, and expected scope.
3. Continue only when the inspector returns parseable JSON with `status: approved` and `specification_gate: open`. A missing input, nonzero exit, unparseable output, mismatch, or unreliable field is `unknown`; a reliable pending gate is `not-approved`. In either case, do not create or materially modify the spec.
4. Resolve the workflow route from the separately supplied route handoff and the approved PRD's `## 工作流分流` section. Continue as `standard` only when both are reliable and agree on `standard`; use the full route for `full` or for a missing or unreliable route. If material technical choices remain after reading the approved PRD and repository evidence, ask one to three independent questions only when no answer changes another question's applicability, options, meaning, priority, or necessity. Ask only one decisive question when its answer selects a branch; preserve partial answers and reassess remaining dependencies.
5. Write the technical spec using [assets/spec-template.md](assets/spec-template.md), preserving its requirements source fields. On the standard route, also create the plan before spec user approval using [assets/plan-template.md](assets/plan-template.md), set `技术规格用户批准: 待批准` and `评审模式: 技术包`, and self-review both as one technical package. On the full route, create only the spec at this stage.
6. For the standard route, dispatch one package reviewer when available. One verdict covers the current spec and plan; fix every finding across either file and re-review the latest package. After approval, the main agent records spec independent-review metadata and plan review metadata, while technical specification user approval remains pending. For the full route, dispatch a fresh read-only spec reviewer and stop after its approval to ask the user to approve the current spec. Apply the known-unavailable reviewer rule without waiting or self-approving.
7. After the user approves the unchanged reviewed spec, the standard route must synchronize the plan metadata from `技术规格用户批准: 待批准` to `已批准`; this metadata-only synchronization does not invalidate the package review. The full route requires that the user approves that reviewed spec before creating the plan, then creates a `评审模式: 逐级` plan and uses a separate plan reviewer. A material spec change invalidates spec review, user approval, and plan review; a material plan-only change invalidates only plan review.
8. Classify every reply before rendering. An `ordinary-clarification` uses local [scripts/render_handoff.py](scripts/render_handoff.py) with the `compact` view and exactly three consecutive lines. A `checkpoint`, blocked reply, spec approval request, plan or document-stage completion, or routing transition uses `full`. A progress-only update is not compact; when uncertain, conservatively use full. Invoke the local script with `handoff_schema: workflow`, preserve its canonical input, and never use partial stdout after failure.
9. After plan approval, revalidate the PRD, spec, plan, and complete fourteen-field handoff. Freeze the canonical English snapshot, then render and validate the authoritative Chinese fourteen-field `full` view before selecting any runtime capability and before session routing. Only after that validation succeeds, if `implementation_gate` is open, select the runtime-exposed skill capability `generating-development-prompts` in the same session for session routing. If that capability is unavailable, report the capability gap without searching installation paths.
10. Full replies, including every routing reply, end with the fixed Chinese fourteen-field status view from the handoff reference unless renderer validation fails. Ordinary clarification ends only with compact status. A renderer failure reports the deterministic Chinese blocker without a partial status view and stops the current automatic handoff. A successful routing response must end with the same fourteen-field snapshot frozen before routing.

## Hard Gates

- Keep an unavailable or inconclusive review blocked. Never replace independent review with author self-review.
- An approved PRD is a hard prerequisite for spec creation or material modification. Re-run its inspection before opening either downstream gate.
- Treat a reliable pending plan-review state in either supported document schema as `not-approved`; treat mixed-schema, malformed, semantic-duplicate, conflicting, unsupported, or unreadable review metadata as `unknown`.
- Before plan creation, require a complete chinese-current spec or a reliable `english-legacy` spec. On the standard route only, its reliable user-approval state may remain pending while the package is created; on the full route it must already be approved. Do not create the plan from a mixed schema, semantic duplicate, missing-field, malformed, or unsupported spec, and keep `implementation_gate` blocked.
- Open `implementation_gate` only while `specification_gate` remains open, spec independent review is approved, the user approved that reviewed spec, and the current plan has real independent approval.
- End every checkpoint, blocked, capability-gap, approval, progress-only, document-stage completion, conservative fallback, or routing reply with the complete fourteen-field handoff record. End only ordinary nonblocking clarification with the compact three-line view. Renderer failure is the explicit fail-closed exception below.
- Do not route when any handoff field is missing or conflicting, or when `implementation_gate` is not open.
- Before selecting the routing capability, validate the Chinese view's field count, field order, exact labels, and that every contextual mapping is complete and unique. On failure, preserve the canonical English snapshot, emit a deterministic Chinese blocker that identifies the unmapped field or failed integrity condition, do not append a status view, do not select the routing capability, and stop the current automatic handoff. Only after the mapping is corrected, retry from the preserved canonical snapshot or its current source.
- Do not implement target code, read sibling skill source, inspect sibling skill installation directories, create or manage a user-visible task/thread, or change external state.
